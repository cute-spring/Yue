# 12-Factor App (十二要素应用) 架构指南

## 什么是 12-Factor App？

**12-Factor App**（十二要素应用）是由 Heroku 的联合创始人 Adam Wiggins 在 2011 年提出的一套构建现代、可扩展、云原生（Cloud-Native）软件即服务（SaaS）应用的方法论和最佳实践。

这套方法论总结了成千上万个云端应用的开发、部署和运维经验，旨在解决应用在云时代面临的核心挑战：
- **声明式配置**：最小化新开发者的上手时间和成本。
- **环境解耦**：在底层操作系统之间提供最大的可移植性。
- **云原生部署**：适合部署在现代云平台（如 AWS、Kubernetes、Docker 等），无需管理服务器和系统。
- **持续交付**：最小化开发和生产环境的差异，实现敏捷的持续部署。
- **弹性扩展**：可以在不需要重大架构改变的情况下实现横向扩展（Scale Out）。

目前，12-Factor 已经成为构建微服务架构和容器化应用（Docker/Kubernetes）的事实标准。

## 十二要素详解与本项目 (Yue) 的映射

### 1. 基准代码 (Codebase)
> **原则**：一份基准代码（如 Git 仓库），多份部署（开发、测试、生产）。
* **含义**：应用的代码应该被版本控制系统（如 Git）管理。一个应用只能有一个代码库。如果多个应用共享同一份代码，那它应该被拆分成一个依赖库。
* **Yue 项目实践**：我们通过 Git 进行版本控制，同一个仓库可以通过不同的配置在本地开发机、CI/CD 管道和最终生产环境中运行。

### 2. 依赖 (Dependencies)
> **原则**：显式声明和隔离依赖。
* **含义**：永远不要隐式依赖系统级的软件包。应该通过依赖清单（如 `requirements.txt`, `package.json`）精确声明所有依赖，并通过虚拟环境或容器进行隔离。
* **Yue 项目实践**：后端使用 `uv`/`pip` 配合 `requirements.txt` 和虚拟环境；前端使用 `npm` 配合 `package.json`。

### 3. 配置 (Config)
> **原则**：在环境中存储配置。
* **含义**：配置（如数据库密码、API Keys、外部服务域名等）在不同部署环境间会有很大差异。代码应当与配置严格分离。最佳实践是将配置存储在**环境变量**中。
* **Yue 项目实践**：我们在 Epic 1 中刚刚完成了这一步！引入了 `pydantic-settings`，彻底解耦了 `global_config.json`，优先从 `os.environ`（环境变量）读取 LLM_PROVIDER 和 API Keys。

### 4. 后端服务 (Backing services)
> **原则**：把后端服务当作附加资源。
* **含义**：数据库、消息队列、SMTP 缓存等都应该被视为附加资源，可以通过 URL 或凭证轻松地在本地服务、第三方服务或云服务之间切换，而无需修改应用代码。
* **Yue 项目实践**：目前的 SQLite 还是本地强耦合的（接下来的 Epic 2: 数据库架构演进，就是要引入 ORM，使其能够像附加资源一样轻松切换到 PostgreSQL 或 MySQL）。

### 5. 构建，发布，运行 (Build, release, run)
> **原则**：严格分离构建和运行阶段。
* **含义**：代码变成运行中的应用必须经过三个严格分离的阶段：
  - **构建**：将代码仓库转化为可执行包（如 Docker 镜像）。
  - **发布**：将构建包与环境配置结合。
  - **运行**：在执行环境中启动进程。
* **Yue 项目实践**：未来的容器化部署将严格遵循这一流程，通过 Dockerfile 进行构建，通过环境变量注入进行发布。

### 6. 进程 (Processes)
> **原则**：以一个或多个无状态进程运行应用。
* **含义**：应用进程必须是**无状态（Stateless）**且**无共享（Share-nothing）**的。任何需要持久化的数据都要存储在有状态的后端服务（如数据库）中。内存缓存只能作为单次请求的短暂优化。
* **Yue 项目实践**：我们的应用逻辑基本是无状态的，但文件上传和本地 SQLite 目前打破了这个规则（它们有状态并存储在本地文件系统）。这是我们后续 Epic 需要解决的核心痛点。

### 7. 端口绑定 (Port binding)
> **原则**：通过端口绑定提供服务。
* **含义**：Web 应用不应该依赖于运行环境中的 Web 服务器（如将应用注入到 Apache 或 Tomcat 容器中）。应用本身应该完全自我包含，并在特定端口上监听 HTTP 请求。
* **Yue 项目实践**：后端通过 `uvicorn` 直接绑定 `8003` 端口提供服务；前端通过 `vite` 绑定端口提供服务。完全符合。

### 8. 并发 (Concurrency)
> **原则**：通过进程模型进行扩展。
* **含义**：应用应该设计成能够通过启动多个相同的进程（横向扩展）来处理更高的负载，而不是仅仅依赖增强单台机器的性能（纵向扩展）或进程内的多线程。
* **Yue 项目实践**：一旦我们完成无状态化（如迁移数据库和文件存储），Yue 就可以通过在 Kubernetes 中增加 Pod 数量来轻松实现无限并发。

### 9. 易处理 (Disposability)
> **原则**：快速启动和优雅终止可最大化健壮性。
* **含义**：应用的进程应该是可以被瞬间启动或终止的（Disposable）。它们应该能够处理突发的宕机，并在收到 SIGTERM 信号时优雅地关闭（清理连接，完成当前请求等）。
* **Yue 项目实践**：我们的 FastAPI 后端天然支持快速启动。

### 10. 开发与线上环境等价 (Dev/prod parity)
> **原则**：尽可能的保持开发，测试，线上环境相同。
* **含义**：缩小时间差异（开发者写完代码立刻部署）、人员差异（写代码的人也参与部署）和工具差异（本地用什么数据库，线上就用什么数据库）。
* **Yue 项目实践**：这也是为什么我们强烈推荐后续通过 Docker Compose 进行本地开发，以确保与线上运行环境的一致性。

### 11. 日志 (Logs)
> **原则**：把日志当作事件流。
* **含义**：**应用不应该试图去写或管理日志文件（如写到本地的 `.log` 文件）**。应用应该将其产生的事件流以纯文本格式输出到 `stdout`。由运行环境（如 Docker、Kubernetes）负责捕获、路由、存档和索引这些日志流。
* **Yue 项目实践**：在刚刚完成的 Epic 1 Phase 2 中，我们确认并强化了这一原则。后端直接使用 `StreamHandler` 输出到 `stdout`，并在生产环境中通过环境变量 `LOG_FORMAT=json` 支持结构化输出流。

### 12. 管理进程 (Admin processes)
> **原则**：后台管理任务当作一次性进程运行。
* **含义**：数据库迁移、数据清理等一次性管理任务，应该与常驻的业务进程在相同的环境中、使用相同的代码基准和配置来运行。
* **Yue 项目实践**：我们计划在 Epic 2 引入 Alembic 进行数据库迁移，迁移脚本将作为一次性任务独立运行。

## 总结
通过刚刚完成的 **Epic 1: 配置与日志演进**，Yue 项目已经在 **第3要素（配置）** 和 **第11要素（日志）** 上完全达到了 12-Factor 的生产级标准。

接下来的架构演进（如数据库分离、文件存储抽象等），本质上都是为了满足 **第4要素（附加资源）** 和 **第6要素（无状态进程）**，最终实现真正的云原生部署。
