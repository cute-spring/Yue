# Agent 管理系统 (创建与修改) UX/UI 改进方案建议

**日期**: 2026-04-18
**状态**: 建议稿 (Proposal)

本文档针对 Yue 项目中 Agent 的创建与修改流程 ( `/agents` ) 提出了一系列深度改进建议，旨在提升开发者体验 (DX)、配置透明度以及 AI 助手的生产力。建议按推荐优先级从高到低排列。

---

## 1. 核心建议：即时沙盒预览 (Live Sandbox Preview)
**优先级**: 🔥 **最高** (Essential for DX)

### 现状
用户在 `AgentForm` 中修改 System Prompt 或工具配置后，必须“保存 -> 退出 -> 切换 Agent -> 发送消息”才能看到效果。这种断裂的反馈循环极大地降低了 Prompt Engineering 的效率。

### 改进方案
- **嵌入式聊天窗口**: 在 `AgentForm` 右侧（或在移动端作为抽屉层）增加一个轻量级的对话沙盒。
- **配置即生效**: 
    - 修改 System Prompt 时，沙盒自动重置上下文。
    - 切换工具或技能时，沙盒内立即反映工具调用的可用性。
- **调试信息显式化**: 在沙盒中，可选显示 Agent 正在思考的过程 (Reasoning) 或命中的技能 (Skill Hits)，帮助用户快速定位 Prompt 的逻辑缺陷。

---

## 2. 能力全景图与冲突检测 (Capability Mapping & Conflict Detection)
**优先级**: 🛡️ **极高** (Clarity & Safety)

### 现状
Agent 的最终权限由 `Tools` (MCP) 和 `Skills` (Markdown) 共同决定。目前采用“取交集”逻辑，且分散在两个不同的配置区块。用户很难一眼看出：我的 Agent 到底能干什么？

### 改进方案
- **能力摘要视图**: 在表单底部或侧边，实时汇总 Agent 的“最终权限清单”：
    - 📂 **文件系统**: 只读 / 读写 (取决于 `read_file`/`write_file` 工具是否开启)
    - 🌐 **网络访问**: 允许 / 禁止 (取决于 `fetch` 类工具)
    - 🛠️ **专业技能**: 已启用 5 个专业 Skill。
- **依赖冲突警告**: 
    - 如果选中的某个 Skill 声明需要 `bash` 工具，但用户在 `Tools` 中禁用了 `bash`，UI 应立即提示：“当前 Skill 缺少必要的 Bash 工具支持，将无法正常运行”。

---

## 3. 智能 Prompt 优化与工具推荐 (Smart Refine & Tool Recommendation)
**优先级**: ✨ **高** (AI-Powered Assistance)

### 现状
`Smart Generate` 只能在从零开始时使用。在已有 Agent 的基础上进行精细化调整时，AI 的辅助作用有限。

### 改进方案
- **Prompt 优化按钮 (Optimize)**: 在 System Prompt 文本框上方增加“一键优化”。
    - **逻辑**: 调用 LLM 分析当前 Prompt，补全缺失的约束条件（如：“请总是以 JSON 格式输出”），并识别出模糊不清的指令。
- **自动工具发现**: 
    - 当用户在 Prompt 中写到“请帮我审阅代码”时，UI 自动弹出提示：“检测到代码审计需求，是否开启 `read_file` 和 `git_diff` 工具？”
- **风险提示**: 针对 `Smart Generate` 生成的工具配置，增加更明显的风险等级标记（READ: 绿色, WRITE: 红色）。

---

## 4. 检索域可视化校验 (Search Scope Real-time Preview)
**优先级**: 🔍 **中** (Utility for RAG)

### 现状
用户在 `Search Scope` 中添加路径（如 `backend/src`）时，不确定该路径在服务器上是否真实有效，也不确定 `File Type Allowlist` 是否配置正确。

### 改进方案
- **路径探测**: 添加路径后，立即调用后端 `ls` 或类似接口，反馈：“匹配到 128 个文件，包含 `.py`, `.md`”。
- **文件白名单模拟**: 实时显示前 5 个将被索引的文件名，帮助用户确认通配符（如 `src/**/*.ts`）是否符合预期。

---

## 5. Prompt 版本快照与对比 (Versioning & Snapshots)
**优先级**: ⏳ **中** (Safety & Experimentation)

### 现状
System Prompt 是 Agent 的灵魂。目前的修改是覆盖式的，如果一次大的调整导致 Agent 表现变差，用户很难回滚到上一个稳定版本。

### 改进方案
- **本地历史快照**: 每次保存时，自动记录上一个版本的 Prompt 摘要。
- **版本对比 (Diff)**: 允许用户查看当前 Prompt 与旧版本的差异。
- **快速回滚**: 一键恢复到 10 分钟前或上一个保存的版本。

---

## 6. 视觉身份与个性化 (Visual Identity & Personalization)
**优先级**: 🎨 **低** (UX Polish)

### 现状
所有 Agent 卡片目前仅以名字首字母作为头像，视觉区分度较低。

### 改进方案
- **AI 头像生成**: 根据 Agent 的 `Name` 和 `Description`，利用 DALL-E 或简单的抽象图形算法自动生成一个独特的视觉图标。
- **色彩主题同步**: 允许为 Agent 选择主题色，该主题色将贯穿整个对话界面的气泡样式和进度条。

---

## 总结
通过上述改进，Yue 的 Agent 管理将从一个简单的“配置编辑器”进化为一个专业的“Agent 开发集成环境 (AIDE)”。**建议首先着手实现第 1 项（沙盒预览）和第 2 项（能力汇总）**，这将为用户带来最直接的效能提升。

---

## 7. 布局与视觉重构建议 (Layout & UI Reconstruction)
**优先级**: 🏗️ **高** (User Interface Polish)

### 现状问题 (Layout "Strange" Points)
- **单列过长**: 目前 `AgentForm` 采用单列堆叠布局，随着功能增加（语音、RAG、技能分组），垂直滚动距离过长，核心配置（Prompt）容易被淹没。
- **空间利用率低**: 在大屏幕上，右侧存在大量留白，而左侧配置项拥挤。
- **层级不清晰**: 基础配置 (Name/Model) 与扩展能力 (Tools/Skills) 的视觉区分度不够。

### 改进方案 (Layout Plan)
- **两栏响应式布局**:
    - **左侧 (60%)**: 专注核心定义 (Identity & Persona)。包含 Name, System Prompt, Intelligence Model。
    - **右侧 (40%)**: 专注能力赋予 (Capabilities)。包含 Tools, Skills, Voice, Search Scope。
- **粘性底部操作栏**: 确保 "Save" 和 "Cancel" 按钮始终可见，无需滚动到底部。
- **分组折叠卡片**: 将非核心功能（如 Voice, Search Scope）默认以折叠卡片形式展示，减少首屏视觉负荷。
- **沉浸式全屏编辑**: 为 `System Prompt` 提供全屏编辑模式，方便编写长篇复杂的指令。

## 8. 改进方案综合分析表 (ROI Analysis)

以下表格从改进价值、开发成本和技术复杂度三个维度对各项建议进行了量化评估，供优先级决策参考。

| 改进项 | 改进价值 (Value) | 开发成本 (Cost) | 复杂度 (Complexity) | 核心技术点 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 即时沙盒预览** | 🔥 **极高** (缩短 90% 调试链路) | 💰 **中** (需复用聊天组件) | 🧩 **高** (需处理未保存状态的 Mock 执行) | 状态同步、沙盒会话管理 |
| **2. 能力全景图** | 🛡️ **高** (消除配置黑盒) | 💰 **低** (纯前端逻辑) | 🧩 **低** (静态逻辑解析) | 权限/工具/技能逻辑求交集 |
| **3. 智能 Prompt 优化** | ✨ **高** (提升 Agent 质量) | 💰 **中** (需增加后端接口) | 🧩 **中** (LLM 提示词工程) | Meta-Prompt 设计 |
| **4. 检索域实时预览** | 🔍 **中** (减少 RAG 失败率) | 💰 **低** (调用现有接口) | 🧩 **低** (Debounce 请求) | 后端文件系统探测接口 |
| **5. 版本快照与对比** | ⏳ **中** (防止误操作) | 💰 **低** (前端 LocalStorage) | 🧩 **低** (文本 Diff 算法) | 差异化渲染 (diff-view) |
| **6. 视觉身份生成** | 🎨 **低** (提升辨识度) | 💰 **低** (调用外部 API) | 🧩 **低** (简单接口对接) | 图片生成/抽象图形算法 |
| **7. 布局与视觉重构** | 🏗️ **高** (提升操作流畅度) | 💰 **中** (需调整组件结构) | 🧩 **中** (响应式适配) | 两栏式 (6:4) 响应式布局 |

**评估说明**:
- **改进价值**: 基于对用户操作频率和痛点缓解程度的预估。
- **开发成本**: 综合考虑前后端工作量及现有组件复用率。
- **复杂度**: 涉及的技术深度、边界情况处理及系统耦合度。
