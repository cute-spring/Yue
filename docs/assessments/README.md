## 📋 文档性质

本目录下的报告（如 `Project_Status_Audit_*.md`）具有以下性质：

### 1. 战略审计 (Strategic Audit)
对已制定的执行计划进行回顾，核实"已承诺"功能与"已交付"代码之间的对齐程度。

### 2. 决策记录 (Decision Record)
记录项目重大优先级调整（如延后特定模块、提前启动新功能）的背景与理由。

### 3. 健康度快照 (Health Snapshot)
客观评估当前工程质量、技术债及模块稳定性。

### 4. 共识基准 (Alignment Baseline)
作为开发者与决策者之间对项目现状及未来走向的统一认知。

---

## 🎯 重要性

- **消除信息不对称**: 确保所有参与者对"哪些已完成、哪些待处理"有清晰、一致的认识
- **风险早期识别**: 通过周期性的进度分析，识别潜在的依赖冲突或技术阻塞项
- **灵活调整优先级**: 项目环境瞬息万变，审计报告提供了基于事实的调整依据，避免盲目执行过时计划
- **历史溯源**: 为未来的项目复盘提供决策链条的原始记录

## 📝 报告结构建议

### 什么是"项目状态审核进度分析报告"？

这类文档（Project Status Audit & Progress Analysis Report, **PSAPAR**）是介于"宏观路线图 (Roadmap)"与"具体执行计划 (Execution Plan)"之间的**战略审计文档**。

#### 核心价值

- **审核 (Audit)**: 核对"已承诺"与"已交付"之间的差距
- **状态 (Status)**: 客观评估各模块的工程健康度（如 Skills 架构是否稳固）
- **分析 (Analysis)**: 当环境或优先级发生变化时（如决定延后 Memory），给出理性的解释
- **对齐 (Alignment)**: 作为开发者与决策者之间的共识基准

#### 推荐结构

```markdown
# Project Status Audit - YYYY-MM-DD

## 执行摘要
一句话说明当前最重大的进度变化。

## 1. 计划执行审计
表格化展示各 P0/P1 项的完成度：
| Epic/计划 | 承诺状态 | 实际状态 | 偏差说明 |
|-----------|----------|----------|----------|

## 2. 偏差分析
解释为什么某些项提前完成，某些项被延后。

## 3. 工程健康度评估
- 技术债情况
- 测试覆盖率
- 代码质量指标

## 4. 决策记录 (Decision Log)
记录本次沟通中达成的共识和新的优先级。

## 5. 下一步优先级
更新后的执行顺序和建议。
```

## 🔄 维护规范

### 审计频率

- **定期审计**: 每 2 周或每个重大 Milestone 结束后
- **触发审计**: 优先级发生显著偏移时
- **年度审计**: 年终项目复盘

### 命名规范

采用 `Project_Status_Audit_YYYYMMDD.md` 格式，便于按时间排序和追踪决策历史。

### 内容闭环

每份审计报告应明确列出：
- ✅ 已审计的计划项
- 📊 执行偏差说明
- 🎯 更新后的下一步优先级

## 🔗 与其他文档的关系

```
战略层：assessments/ (你在哪里)
   ↓
战术层：plans/INDEX.md (总控看板)
   ↓
执行层：plans/*.md (具体计划)
```

- **assessments/**: 决定"做什么"和"为什么做"
- **plans/INDEX.md**: 追踪"正在做什么"和"完成度"
- **plans/*.md**: 指导"具体怎么做"

## 🤖 AI 协作指南

### 生成审计报告

使用以下 Prompt 驱动 AI 生成审计报告：

```
请阅读 docs/plans/INDEX.md 和所有进行中的计划文档，
生成一份项目状态审计报告 (Project Status Audit Report)，
包含：
1. 各 Epic 的完成度评估
2. 执行偏差分析
3. 下一步优先级建议
```

### 使用审计报告

审计报告完成后：
1. 更新 [plans/INDEX.md](../plans/INDEX.md) 中的 Epic 状态
2. 根据新的优先级调整执行顺序
3. 在对应计划文档中添加审计引用
