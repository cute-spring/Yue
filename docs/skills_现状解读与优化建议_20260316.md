# Skills 专题：现状解读与优化建议（2026-03-16）

## 1. 结论摘要

- 当前项目的 skills 体系已具备可运行的工程闭环：技能规范、注册加载、运行时选择、提示词注入、工具权限收敛、前端交互与效果观测均已落地。
- 当前阶段重点不在“有没有”，而在“选得更准、注入更省、治理更强”。
- 建议优先推进三件事：路由精度提升、校验器增强、可观测字段补全。

## 2. 设计与实现现状

### 2.1 启动与注册

- 服务启动时会初始化技能目录 `backend/data/skills`，并加载到全局注册表 `skill_registry`。
- Skills API 已挂载为 `/api/skills`，支持列表、详情、重载与选择能力。

参考：
- `backend/app/main.py`
- `backend/app/api/skills.py`

### 2.2 技能模型与格式

- 核心数据模型 `SkillSpec` 已覆盖：
  - 基础信息：`name/version/description/capabilities/entrypoint`
  - 运行约束：`constraints.allowed_tools`
  - 环境治理：`requires/os/install`
  - 运行态信息：`availability/missing_requirements`
  - 内容分段：`system_prompt/instructions/examples/failure_handling`
- 解析器支持两种形态并存：
  - 历史单文件 `.md`
  - 目录化 `SKILL.md`
- 验证器当前主要校验必填项与 entrypoint 对应分段存在。

参考：
- `backend/app/services/skill_service.py`
- `backend/data/skills/ppt-expert/SKILL.md`
- `backend/data/skills/release-test-planner.md`

### 2.3 运行时选择、注入与权限

- 路由策略：
  - 先按 agent 的 `visible_skills` 做可见范围过滤（fail-closed）
  - 支持显式请求技能（`requested_skill`）与消息内自动推断
  - 支持 manual / auto 模式与会话绑定技能
- 提示词注入：
  - 选中技能后拼装 `persona + active skill + instructions`
  - 支持 always 技能附加注入
  - 支持按需 full load（lazy）
- 工具权限：
  - 最终工具集合通过 `agent.enabled_tools ∩ skill.allowed_tools` 收敛

参考：
- `backend/app/api/chat.py`
- `backend/app/services/skill_service.py`
- `backend/app/services/chat_service.py`

### 2.4 前端使用链路

- Agent 配置页支持 `skill_mode` 与 `visible_skills` 编辑。
- Chat 页在 manual 模式下支持显式选 skill，并将 `requested_skill` 随请求发送。
- SSE 事件 `skill_selected` 驱动 “Active Skill” UI 展示。
- 技能不可用时可在下拉中显示 unavailable 与缺依赖摘要。

## 3. 当前优势与边界

### 3.1 优势

- 兼容迁移路径清晰：旧 `.md` 与 `SKILL.md` 可并存。
- 工具权限边界明确：交集策略避免技能越权放大。
- 运行时稳定性较好：会话绑定与 fallback 机制完备。
- 观测基础较扎实：有事件、有聚合、有日报。

### 3.2 主要边界

- 路由精度仍以规则打分为主，复杂语义场景存在误选空间。
- 校验器深度偏浅，尚未覆盖 schema 语法与策略级约束。
- 注入策略虽已做摘要/lazy，但仍缺显式 token 预算治理。

## 5. 优化建议（按优先级）

### P0（立即）

- 统一 `allowed_tools` 语义，消除“无 constraints 时是否等价于全禁工具”的路径差异，确保 chat 与 select API 行为一致。

### P1（高 ROI）

- 提升路由精度：
  - 在规则打分前后引入轻量语义检索或重排
  - 输出路由置信度到效果事件，支持阈值数据化调优
- 增强校验器：
  - 增加 `inputs_schema/outputs_schema` 语法校验
  - 增加 constraints 合法性与危险声明阻断
  - 增加 section 最小完备性校验

### P2（中期）

- 注入预算化：
  - 为 always 技能、summary、附加 instructions 设置 token 预算
  - 超预算时执行可解释裁剪并记录原因
- 观测增强：
  - 增加 `rejected_reason/switch_reason/tool_intersection_size` 等字段
  - 支持直接回答“为何没选中/为何切换/为何回退”

### P3（持续）

- 建立技能 CI 基线：
  - 静态 lint + 回放测试 + 依赖探测
  - 每个技能至少一条“触发成功”和一条“回退安全”用例
- 推进技能分发能力：
  - 内部 registry、安装/升级/回滚与兼容性检查

## 6. 推荐执行顺序（6周）

- 第 1-2 周：修复 P0 + 落地 P1 的路由置信度与校验器扩展
- 第 3-4 周：落地 P2 的 token 预算与观测字段增强
- 第 5-6 周：搭建技能 CI 基线，并启动 registry 最小可用版本设计

## 8. P0 执行清单（可直接开工）

### 8.1 目标与范围

- 目标：修复 `allowed_tools` 在 chat 运行时路径与 `/api/skills/select` 路径的语义不一致问题。
- 范围：仅修复工具策略一致性，不改变 skill 选择策略（manual/auto）、评分逻辑和 prompt 注入策略。
- 非目标：不引入新路由算法、不修改前端交互、不扩展 skill schema 字段。

### 8.2 语义契约（先定规则再改代码）

- 约定 A：`skill.constraints.allowed_tools` 缺失或为 `None` → 表示“技能不额外限制工具”，最终工具集等于 `agent.enabled_tools`。
- 约定 B：`skill.constraints.allowed_tools == []` → 表示“技能显式禁止所有工具”，最终工具集为空数组。
- 约定 C：`skill.constraints.allowed_tools == [x, y, ...]` → 表示“技能白名单限制”，最终工具集为 `agent.enabled_tools ∩ allowed_tools`。
- 约定 D：以上语义在 chat 路径与 `/api/skills/select` 路径必须完全一致。

### 8.3 影响面清单（文件级）

- `backend/app/services/skill_service.py`
  - `MarkdownSkillAdapter.to_descriptor`：确保无 constraints 时输出不触发“误判为空列表”。
  - `SkillPolicyGate.check_tool_intersection`：保持三态语义稳定。
- `backend/app/api/chat.py`
  - 技能命中后 `allowed_tools` 获取与 `final_tools_list` 计算逻辑需与 API 路径一致。
- `backend/app/api/skills.py`
  - `/select` 与 `/tool/select_runtime_skill` 的 `effective_tools` 计算语义保持一致。
- `backend/tests/*`
  - 新增或补齐语义一致性测试。

### 8.4 实施步骤（推荐半天到一天）

1) 先写测试（红灯）
- 补三组核心断言：
  - 无 constraints（或 `allowed_tools=None`）→ 保留 agent 全工具。

  - `allowed_tools=['a','b']` → 返回交集。
- 在两条路径都断言：
  - API：`/api/skills/select`（必要时含 `/tool/select_runtime_skill`）

3) 回归验证
- 运行新增单测。
- 运行 skill 相关回归（foundation/runtime/api）。
- 确认前端 e2e 关键场景不受影响（至少 manual 选 skill 与 fallback 场景）。

4) 发布与回滚准备
- 若可行，增加临时开关（示例：`skill_tool_policy_v2_enabled`）用于快速回退。
- 记录发布说明：本次是“语义一致性修复”，非功能扩展。

### 8.5 验收标准（Definition of Done）

- 一致性：同一 skill 配置在 chat 与 `/api/skills/select` 返回的 `effective_tools` 一致。
- 覆盖率：三态语义（None/[]/list）均有自动化测试覆盖。
- 稳定性：manual/auto 选 skill 行为不回归，fallback 逻辑不回归。
- 可观测性：至少可从日志或事件确认本轮使用的工具策略模式与结果数量。

### 8.6 风险与对策

- 风险 1：历史 skill 未显式 constraints，修复后工具可用性变化。
  - 对策：先扫描现有技能配置，发布前做样本回放。
- 风险 2：修复影响 chat 与 API 之外的隐式调用点。
  - 对策：集中复用 `SkillPolicyGate`，避免分散逻辑。
- 风险 3：测试只覆盖 API 未覆盖 stream 真路径。
  - 对策：补 chat runtime 集成测试，断言传给 `tool_registry` 的最终工具集。

### 8.7 上线后 24 小时观察项

- `fallback_rate`：是否异常上升。
- `tool_call_success_rate`：是否下降。
- `skill_selected_rate`：是否出现异常波动。
- 若任一指标超过预设阈值，执行开关回退或快速修复。
