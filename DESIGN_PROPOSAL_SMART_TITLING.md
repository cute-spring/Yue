# 对话元信息轻量方案（Title + Summary）

## 1. 背景
当前会话标题在首条用户消息写入时由截断生成，语义质量有限。
- 现有逻辑位置：[chat_service.py](backend/app/services/chat_service.py#L529-L539)

本方案目标是：在不增加主链路复杂度的前提下，快速提升标题与摘要质量。

## 2. 方案结论（最终版）
采用一个**系统级 Meta LLM**，专门负责生成：
- `title`：会话标题
- `summary`：会话摘要（短文本）

主对话模型继续只负责回答；Meta LLM 走异步，不阻塞聊天流。

## 3. 最小可落地设计

### 3.1 单一能力入口
后端提供一个统一方法，例如 `generate_session_meta(...)`，内部根据任务类型生成：
- `task=title`
- `task=summary`

这样只维护一套调用链路和一套错误处理，减少复杂分支。

### 3.2 系统级配置
在全局配置增加专用字段：
- `meta_provider`
- `meta_model`
- `meta_timeout_ms`
- `meta_max_tokens`
- `meta_enabled`（功能开关）

默认使用低成本、低延迟模型，和主对话模型解耦。

### 3.3 触发时机（保持简单）
- **Title**：仅在首轮问答后异步生成 1 次。
- **Summary**：先不做复杂实时更新，采用“按需生成”：
  - 可在会话结束后生成一次，或
  - 用户打开会话历史详情时生成一次。

### 3.4 失败降级
- 超时/失败：保留原始标题，不影响主对话。
- 不重试或最多重试 1 次，避免后台任务膨胀。
- 不向用户弹错误提示，保持体验稳定。

## 4. Prompt 最小模板

### 4.1 Title Prompt
```text
你是对话标题生成器。
请根据用户首条消息与助手首段回复，生成一个简洁标题。
要求：
1) 输出仅标题，不要解释；
2) 中文控制在 2-8 字，英文控制在 3-5 词；
3) 去掉客套话，保留任务核心意图。
```

### 4.2 Summary Prompt
```text
你是对话摘要生成器。
请输出 1-2 句摘要，说明用户诉求与当前结论。
要求：
1) 输出纯文本，不要列表；
2) 避免细节噪音，突出可检索关键词；
3) 80 字以内（英文 40 词以内）。
```

## 5. 实施步骤（精简）
1. 增加 Meta LLM 全局配置与开关。
2. 新增统一 `generate_session_meta` 服务方法。
3. 在首轮 assistant 回复完成后触发 `title` 异步任务。
4. 在会话读取或结束时触发 `summary` 生成（二选一，先选更简单的一种）。
5. 前端沿用现有 history 刷新机制，不新增复杂事件协议。

## 6. 验收标准
- 首轮后标题可从截断文本升级为语义标题。
- 主对话首字延迟与总时延无明显变化。
- Meta 生成失败不影响会话可用性。
- 成本可控，且明显低于主对话模型调用成本。
