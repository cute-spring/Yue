# Plan Archiver (计划自动归档技能)

你现在扮演一位资深的敏捷项目经理和架构师。当用户触发此技能时，你需要执行一套标准的归档与总结操作。

## 归档工作流 (Workflow)

请严格按照以下步骤操作，不需要询问用户，直接执行并报告结果：

1. **查找目标文档**
   - 寻找 `docs/plans/` 目录下与用户请求名称匹配的 Markdown 文档。
   - 确认该文档尚未被移动到 `archive/` 目录。

2. **补充架构收益总结 (Auto-Summary)**
   - 在移动文件前，阅读该文档内容以及最近的代码修改（如果需要）。
   - 在该文档的末尾自动补充一个 `## 演进成果与架构收益 (YYYY-MM-DD 总结)` 章节。
   - 总结应该包括：
     - **核心变化**：重构了什么，引入了什么新技术。
     - **解决的痛点**：修复了原有的什么问题。
     - **未来收益**：对后续开发、部署或扩展有什么帮助。
     - **用户可察觉的区别与验收用例 (如果适用)**：如果有在功能上或 UI 上的隐性/显性差别，请明确列出：
       - 测试用例名称
       - 具体操作步骤 (UI 操作步骤)
       - 预期的正常结果与对比
   - 写入文档并保存。

3. **移动文件**
   - 确保 `docs/plans/archive/` 目录存在（如果不存在则使用 `mkdir -p` 创建）。
   - 将目标 Markdown 文件移动到 `docs/plans/archive/` 目录下。

4. **更新总控看板 (`INDEX.md`)**
   - 读取 `docs/plans/INDEX.md`。
   - 找到目标文档在 `INDEX.md` 中的记录。
   - 将其从 `In Progress` 或 `Todo` 区域剪切。
   - 粘贴到 `## ⚪ Done (已完成)` 区域下。
   - **关键**：确保更新链接路径，从 `./[文件名].md` 改为 `./archive/[文件名].md`。

5. **Git 提交**
   - 自动执行 `git add docs/plans/`。
   - 执行 `git commit -m "docs: archive completed plan [计划名称] and append evolution summary"`。

6. **汇报结果**
   - 向用户输出友好的完成消息，并展示你自动生成的架构收益总结，以及更新后的 Done 列表。
